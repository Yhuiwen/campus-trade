# Java 后端实习面试讲解稿：校园二手交易平台

## 1. 30 秒项目介绍

这个项目是我基于 Spring Boot 3 和 Vue 3 完成的校园二手交易平台，主要面向校园内用户进行闲置物品发布、浏览、下单、评价和后台审核。后端负责用户认证、商品管理、订单状态流转、信用评价、收藏浏览记录、Redis 缓存和规则版交易助手等功能。项目中我重点实践了 Spring Security + JWT、MyBatis-Plus、MySQL 表设计、Redis 缓存、前后端接口联调和后台管理系统开发。

## 2. 2 分钟项目介绍

我做这个项目是因为校园内二手交易常见问题比较明确：商品信息需要审核，买卖双方需要基本信用参考，热门商品和商品详情访问频繁，同时还要避免同一件商品被多人重复购买。

后端采用 Spring Boot 3，分为 controller、service、mapper、entity、dto、vo、config、security 等层。用户登录后通过 JWT 访问受保护接口，普通用户可以发布商品、编辑自己的商品、收藏、下单和评价，管理员可以审核商品、管理用户和查看数据看板。商品发布后先进入 `PENDING`，管理员通过后变成 `ON_SALE`，买家下单时通过条件更新把商品锁定为 `LOCKED`，卖家确认完成后订单变为 `COMPLETED`，商品变为 `SOLD`。

我主要负责后端接口、数据库表设计、状态流转、权限控制、Redis 缓存和前后端联调。比较能体现后端能力的地方是：用 Spring Security + JWT 做无状态认证；用 service 层校验数据归属防止越权；用事务和条件更新控制订单与商品状态一致性；用唯一索引保证收藏、浏览历史和评价的幂等；用 Redis 缓存热门商品和商品详情，同时支持 Redis 异常时降级回源数据库。

## 3. 技术亮点

- Spring Security + JWT 登录认证，区分 `USER` 和 `ADMIN`。
- 商品审核与状态流转：`PENDING`、`ON_SALE`、`LOCKED`、`SOLD`、`OFF_SHELF`、`REJECTED`。
- 订单状态控制：`CREATED`、`CANCELLED`、`COMPLETED`。
- 信用评价机制：买家完成订单后评价卖家，星级影响信用分。
- 收藏与浏览历史：组合唯一索引保证幂等，浏览历史重复访问只更新时间。
- Redis 缓存：缓存热门商品和商品详情，商品状态变化时主动失效。
- 规则版交易助手：提供发布建议、价格评估、风险分析、个性化推荐和问答。
- 后台数据看板：统计用户数、商品数、订单趋势、成交金额、商品状态和分类分布。

## 4. 高频追问与回答

1. JWT 登录流程是什么？
   用户提交用户名密码，后端用 Spring Security 的密码编码器校验 BCrypt 密文，校验通过后生成 JWT。前端保存 token，后续请求通过请求头携带，过滤器解析 token 并写入认证上下文。

2. Spring Security 如何区分 USER 和 ADMIN？
   用户表有 `role` 字段，登录后封装为 `ROLE_USER` 或 `ROLE_ADMIN`。配置中 `/api/admin/**` 使用 `hasRole("ADMIN")` 限制。

3. 如何防止用户修改别人的商品？
   `GoodsService.owned` 会根据商品 id 查询商品，并校验 `seller_id` 是否等于当前用户 id，不一致就返回 403。

4. 为什么商品需要审核？
   二手交易平台需要控制违规内容和低质量信息，所以商品发布或编辑后进入 `PENDING`，由管理员审核后才能在首页展示。

5. 商品状态和订单状态怎么设计？
   商品状态描述商品是否可售或审核情况，订单状态描述交易单生命周期。两者分离后，商品审核、锁定、售出和订单取消、完成都更清晰。

6. 如何避免重复购买？
   创建订单时先要求商品是 `ON_SALE`，再执行条件更新：只有当前仍是 `ON_SALE` 才能改为 `LOCKED`。如果更新行数为 0，说明已被别人锁定或状态变化。

7. 为什么要有 LOCKED 状态？
   `LOCKED` 表示商品已进入交易中，阻止其他买家继续下单。如果买家取消订单，再恢复为 `ON_SALE`。

8. 订单完成后为什么才允许评价？
   评价应该基于真实交易结果，未完成订单评价会影响信用分的可靠性，所以 service 层校验订单必须是 `COMPLETED`。

9. 如何避免重复评价？
   `review.order_id` 是唯一键，同时代码查询同一订单是否已有评价，双层保证一笔订单只评价一次。

10. 信用分怎么变化？
    5 星 +2，4 星 +1，3 星不变，2 星 -2，1 星 -5，并限制在 0-150。

11. 收藏功能如何保证幂等？
    `favorite` 表有 `user_id + goods_id` 唯一索引，重复收藏不会生成多条记录。

12. 浏览历史为什么重复浏览只更新时间？
    这样能保留“最近浏览”的排序价值，又不会让同一商品刷出多条历史记录。

13. Redis 缓存了哪些数据？
    主要是 `goods:hot` 热门商品列表和 `goods:detail:{id}` 商品详情，默认 10 分钟过期。

14. 商品更新后如何保证缓存一致性？
    商品创建、编辑、下架、订单创建、取消和完成时都会调用缓存清理，删除商品详情和热门商品缓存。

15. Redis 挂了核心功能是否能用？
    可以。`CacheService` 捕获 Redis 读写异常，失败时回源数据库查询，不阻塞主流程。

16. 图片上传怎么处理？
    后端限制 jpg/jpeg/png，文件保存到配置的 `UPLOAD_DIR`，并通过静态资源映射暴露 `/uploads/{filename}`。

17. 前端如何携带 JWT？
    Axios 请求封装会读取本地 token，并在请求头中携带，后端过滤器负责解析和鉴权。

18. 跨域如何解决？
    后端通过 `CORS_ALLOWED_ORIGINS` 配置允许来源；开发环境下前端也通过 Vite 代理 `/api` 和 `/uploads`。

19. 接口返回格式如何统一？
    后端使用 `ApiResponse` 包装响应，成功返回 `code`、`message`、`data`。

20. 数据看板统计哪些指标？
    包括用户数、商品数、订单数、成交金额、商品状态分布、分类商品数量和最近 7 日订单趋势。

21. 推荐功能的数据来源是什么？
    主要来自用户收藏和浏览历史，并结合在售商品、分类偏好和热门商品进行规则排序。

22. 规则版 Agent 和真实大模型有什么区别？
    当前 Agent 是本地规则实现，不依赖外部 API，结果可解释但泛化能力有限。代码预留了 `AgentModelProvider`，后续可以接入真实模型。

23. 如果数据量变大，商品搜索怎么优化？
    可以给标题、分类、状态建立更合适的索引，引入 Elasticsearch 或 MySQL 全文索引，并对分页和排序字段做优化。

24. 如果多人同时买一个商品，如何处理并发？
    用数据库条件更新保证只有一个请求能把商品从 `ON_SALE` 改成 `LOCKED`，再在事务中创建订单。

25. 为什么订单完成和商品售出要放在事务里？
    订单状态和商品状态必须一致，事务能避免订单已完成但商品没售出，或商品售出但订单没完成的中间状态。

26. 管理员为什么不能被封禁？
    `AdminService` 中对管理员封禁做了保护，避免误操作导致后台无人可用。

27. 项目还有哪些可以改进？
    可以增加站内消息、支付或担保流程、真实大模型 Agent、搜索服务、更多自动化测试，以及更细粒度的权限和审计日志。
